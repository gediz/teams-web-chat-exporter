- [ ] **base64 → binary image transport (the "renovation").** Image bytes cross content→background as base64 dataUrl strings. Messaging is copy-only JSON (confirmed by research): no `ArrayBuffer`/`Blob` transferables, and the per-message cap is a hard **64 MiB measured on the JSON-serialized payload**, identical for `sendMessage` and `Port.postMessage`. Bytes round-trip wastefully — for bearer images the SW fetches → base64 → content → base64 back inside the streamed `messages` batches (a double crossing) — costing ~1.33x size, ~2.74x peak heap, forcing the per-image `MAX_IMAGE_BYTES` cap, and capping the whole pipeline's memory. **Stopgap shipped** (`fidelity-fixes-batch1`): byte-aware streaming batching + per-message sub-chunk reassembly (`content.ts` peels an oversized message's largest images into standalone `attachment-chunk` port messages; `background.ts` reattaches by message+attachment index) so a full-res image-dense chat can never overflow one port message and lose the chat — but the bytes still cross the wall. **Real fix:** stop sending image bytes across the content↔background boundary at all. Cleanest direction — have the **background keep the bytes it already fetches** (it already runs the `FETCH_BLOB` image fetch), so the content script sends only references. For images only the content script can fetch (cookie/page-world paths), the **only** viable content→background shared store is `chrome.storage` with the `unlimitedStorage` permission: the content script's IndexedDB / OPFS / `blob:` URLs live in the **Teams page origin**, invisible to the extension SW, so that whole family is ruled out. This removes the base64 inflation, lets the per-image cap rise/disappear, and is the **same** transport foundation real document-download needs. Firefox MV2 (structured-clone messaging, no 64 MiB cap, no offscreen document) needs the bytes-stay-in-background design too but doesn't hit the same wall. Deferred; bundle with document-download.
- [ ] **Full-res PDF sweet spot.** With the full-resolution-images opt-in, the PDF embeds the full PNG via `doc.embedPng` with NO downsampling (`pdf.ts`), yet draws it capped at `IMAGE_MAX_H=320pt` (~500px on screen). So a 3600px screenshot is stored at full pixel data but displayed as a thumbnail — PDF size grows ~10-50x per image for zero visual benefit, plus a JPEG(imgo)→PNG(imgpsh_fullsize) format shift. v1 ships full-res-affects-PDF and documents it ("much larger files"). Later: downsample/re-encode the image for the PDF (target ~1500px JPEG q90 — JPEG also skips pdf-lib's slow PNG decode), which needs a local canvas resize (MV3 offscreen / MV2 background-page) or a second imgo fetch for the PDF. That's the deferred PDF-optimize.

- [ ] **Document download: skip "request access" pages for inaccessible files.** The Files feature is fire-and-forget by necessity (`downloadAttachments` calls `chrome.downloads.download` without awaiting): on a remote URL, `download()` doesn't resolve until the transfer starts (~1s each), so awaiting it serialized dispatch at the network rate and a few-hundred-file set outran the MV3 service-worker lifetime — it was evicted mid-phase and the export never wrote a history row (verified chrome/68). Firing without awaiting makes the export complete reliably (chrome/69: 221 queued, history row written), but we can no longer inspect each response, so a file the current user can't open saves the SharePoint "request access" HTML page (~180 KB) under the file's name. On a busy chat that can be most of the candidates. Re-add cleanup WITHOUT reintroducing the await: an event-driven `chrome.downloads.onChanged` listener that, as each attachment download completes, checks `mime` (text/html for non-markup, or an access-denied/login `finalUrl`) and `removeFile`+`erase`s it. Best-effort only (stops if the SW is evicted) and must NEVER delete a real `download.aspx`-routed markup file — for those, delete only on an explicit access-denied URL, never on a bare text/html mime. The removed per-file `classifyDownload` (git history on `feat/attachment-download`) has the working mime/URL detection to lift from.
- [ ] **Document download flow: "export complete" fires before the files finish.** The history row and terminal `complete` status are written as soon as the export zip is saved and the attachment downloads are queued, but the per-chat attachment downloads keep streaming in the browser afterward (some are large), so the user sees the export "done" while files are still arriving and had to manually cancel big ones for faster apparent completion (chrome/69). Improve the flow: surface attachment-download progress/state in the status or history (a "downloading files" sub-state that resolves when the `chrome.downloads` items complete, or a per-file completed count), or hold the terminal status until attachments finish. Must NOT block the export handler on download completion — that is exactly what broke reliability — so any progress tracking has to be event-driven (`onChanged`) and tolerant of SW eviction.
- [ ] **Document download: note an aborted export in FAILURES.txt.** When the user aborts a running export, every in-flight attachment download is cancelled, and each lands in FAILURES.txt's "cancelled" section (`verifyDownloadOnChanged` treats an interrupt with no reason as a cancel). A full abort can therefore produce a long "cancelled" list with no sign it was a deliberate whole-export stop. Add a clearer signal: a one-line "the export was stopped before all files finished" note, or a distinct heading, so a deliberate abort reads differently from a few individually-cancelled files. The abort state lives on the export's `AbortSignal` in `downloadAttachments`, but the verifier runs decoupled in `downloads.onChanged`, so threading the signal through (or stamping a per-export "aborted" flag the writer reads) is the open design question.
- [ ] **Firefox: files on a SharePoint host other than the user's own fail to download.** A file the user has direct access to, but which lives on a *different* SharePoint host than their own personal site (a colleague's OneDrive in another region/tenant), downloads correctly on Chrome yet consistently comes back as a "request access" page on Firefox. Reproduced across 6 consecutive Firefox runs with Enhanced Tracking Protection OFF: the same file is a real download on Chrome every time and a request-access failure on Firefox every time, while the user's OWN files and genuinely-no-access files behave identically on both browsers. So it is not intermittent and not an ETP setting. Likely cause: the Firefox MV2 background download context does not carry the FedAuth cookie for the other SharePoint host (whether Firefox never established that host's session, or `browser.downloads.download` does not attach a cross-host cookie that `chrome.downloads` does, is unconfirmed). Investigate whether priming the session (a background fetch to the other host to establish FedAuth before dispatch) recovers them; if not, document it as a Firefox limitation for cross-host/cross-region shared files.

### Post-v1.4 fixes (on `main`, unreleased at time of writing)

- [x] **Teams Free image embedding.** Inline AMS images (URLs in message HTML matching `*.asm.skype.com/v1/objects/...`) now embed in HTML/PDF on Teams Free. Work/School routes through `*.asyncgw.teams.microsoft.com` (the authenticated image proxy, requires userId+region+IC3 token); Teams Free has no consumer equivalent of that proxy, so we fetch the AMS URL directly with `credentials: 'include'` plus the Skype JWT in the `authentication` header. A one-time `POST /v1/skypetokenauth` exchange establishes the `skypetoken_asm` cookie that the AMS host needs alongside the JWT. Path-gated to Teams Free (`location.hostname.includes('teams.live.com')` AND `*.asm.skype.com/v1/objects/` URL pattern) — Work/School proxy path unchanged.
- [x] **TXT/CSV attachment-only message placeholders.** Image pastes and file drops produced silent blank rows (`[ts] author: ` with no body). New `summarizeAttachments` helper in `builders.ts` inspects each empty-text message's attachments and emits a one-line summary like `[image]`, `[image: name.png]`, `[file: name.bmp]`, `[GIF]`, `[link: title]`. Used by both `toCSV` (in the readable text column; the `attachments_json` column still carries the structured data) and `toPlainText`. JSON unchanged (full structured data already in `attachments[]`); HTML unchanged (renders inline images via `contentHtml`).
- [x] **Empty PDF size fix (HarfBuzz).** Empty documents (0 messages) produced ~7 MB PDFs because `subsetFont` bailed when given an empty codepoint set, and the caller's fallback embedded the FULL font (CJK font alone is 6 MB). Fixed by adding U+0020 (space) to an otherwise-empty input: HarfBuzz produces a valid 1-glyph subset that pdf-lib embeds happily, and empty PDFs drop from ~7 MB to ~20 KB. Non-empty exports already had non-empty codepoints so they were unaffected.
- [x] **NO_HISTORY.txt for empty chats in bundles.** Replaced per-chat folders for 0-message chats with a single `NO_HISTORY.txt` at bundle root listing folder + conversationId. Distinct from `FAILURES.txt`: these aren't failures, the API ran fine. Saves ~7 MB per empty chat (no PDF/HTML/CSV/JSON/TXT files generated). All-empty bundles save the txt directly without a zip wrapper, mirroring the existing all-failed UX.
- [x] **System event author resolution on Teams Free.** Teams Free uses Skype consumer IDs (`8:live:<id>`, `8:<id>`) in `<initiator>` / `<part identity>` content; the converter's resolver only matched UUID-style MRIs, so every system event rendered as "(unknown user)". Two fixes: (1) `extractMri` now matches `/contacts/(8:[^/?#]+)` so the full URL form Teams Free uses (`https://msgapi.teams.live.com/v1/users/ME/contacts/8:<id>`) gets resolved correctly; (2) new `loadTeamsFreeProfiles()` reads the `Teams:profiles:*` IDB store (mri → displayName) and merges into the converter's mri map. Path-gated to `teams.live.com` so Work/School never reads the consumer profile DB. Memoised at module scope with a 60-second TTL — a 14-chat bundle pays one IDB roundtrip instead of 14.
- [x] **Teams Free API auth.** The `teams.live.com` chat-service proxy authenticates with `authentication: skypetoken=<JWT>` (note the literal `skypetoken=` prefix and lowercase header name). The required short-lived JWT is encrypted in localStorage at `tmp.auth.v1.<cid>.Discover.SKYPE-TOKEN` (AES-256-CBC, key in clear at `tmp.auth.v1.GLOBAL.ExportedEncryptionKey.ExportedEncryptionKey.item.exportedKey`). `api-client.ts` now decrypts and uses it directly, skipping the network authz round-trip when the cache is fresh. Per-page token re-read no longer substitutes the IC3 MSAL token (which the consumer proxy 401s on). Result: groups, channels, modern 1:1s (`@thread.v2`, `uni01_*@thread.v2`) and `@thread.skype` group chats now export from Teams Free. Legacy Skype-imported `@oneToOne.skype` 1:1s still produce empty exports — see the open TODO above.
- [x] **0-message API result is a valid empty export, not a failure.** Previously a Teams Free legacy Skype-imported chat (server returns `messages: []` because Microsoft never migrated the history) fell through to DOM-scroll fallback, which is correctly disabled in multi-chat bundles, landing the chat in `FAILURES.txt` with a misleading "API scrape failed" reason. Now an empty raw-API result produces a 0-message export per format (the per-format converters already handle `messages = []` cleanly). Filter-everything-out cases (raw had messages, all dropped by the system/reaction filters) keep the existing DOM-scroll fall-through behaviour.
- [x] **Outer-zip OOM fix.** `buildZipAsync` now resolves a `Blob` constructed directly from the chunk array; downloads consume the Blob via `URL.createObjectURL`. The previous contiguous-`Uint8Array` finalize plus `new Blob([uint8])` in `binaryToDownloadUrl` doubled peak memory at the worst moment and blew Firefox's allocation cap on a 2.5 GB / 289-chat bundle. Verified end-to-end on the same workload.
- [x] **Smart per-extension deflate level.** Already-compressed inputs (`.zip`, `.pdf`, `.jpg`, `.png`, `.gif`, `.webp`, `.mp3`, `.mp4`, `.webm`, `.gz`, `.7z`, `.rar`, `.bz2`, `.xz`) get level 0 (store); raw text gets level 6. Same final size, ~3× faster outer-zip step on real bundles.
- [x] **Bounded retry on transient 403 / 5xx in `fetchPageWithRetry`.** 3 attempts at 1s / 2s / 4s. The 5-attempt 429 policy is unchanged.
- [x] **`ScrapeOptions.noDomFallback` for multi-chat bundles.** Bundle iteration sets it per-chat. The content script reports a clean error instead of falling back to DOM scrolling, which would scrape whichever chat is currently visible in the user's tab — not the target chat. Failed chats land in `FAILURES.txt`.
- [x] **All-failed bundle UX.** When 0 chats succeed, skip the .zip wrapper and save `TeamsExport_bundle_<stamp>_FAILURES.txt` directly. New `HistoryEntry.kind = 'failed'` renders with an amber badge and "all failed" pill in the popup history; Open / Show in folder still work because the file is real on disk.
- [x] Picker now shows a clear "Open Teams web first" message when the popup opens on a non-Teams tab (instead of the misleading "No matches" empty-state).
- [x] Auto-default selection (the active chat) seeds at most once per popup open. Once the user touches the selection, later refresh phases never reinstate the original auto-pick.
