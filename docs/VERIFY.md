# Verifying this build

What you can check about the published extension, and what no one can. Honest about both.

## The limit, first

The code is public and you can read all of it. What you cannot do, for any browser extension from any developer, is cryptographically prove that the bytes a store auto-installs equal this source. The stores hold the signing keys and repackage every upload: Chrome rebuilds your upload into a signed CRX and adds an integrity manifest, Firefox re-packs and signs the XPI. So the installed package is signed by Google or Mozilla, not by us, and its whole-file hash will never match a hash we publish. Trust ultimately roots in the store signature. What we give you is a way to check that the files inside the published package match a build from this source.

## Firefox gives the strongest check

Mozilla requires the full source for the add-on and a reviewer rebuilds it, confirming it matches the shipped package with no differences. That is an independent party verifying source against the shipped code. You can also check it yourself: download the XPI from the AMO listing, unzip it, and compare each file (ignoring the added `META-INF/` signature) against a build from the tagged commit.

## Chrome and Edge: compare the files yourself

1. Download the live extension with Extension Source Viewer (https://github.com/Rob--W/crxviewer), or read the installed files from disk (`chrome://extensions` in Developer mode shows the path).
2. Unzip it and ignore the store-added `_metadata/` folder.
3. Compare each remaining file's SHA-256 against a build from this repo at the matching release tag. They should match.

## Build provenance (SLSA)

Each GitHub release carries SLSA Build Level 2 provenance for the zips, generated by `.github/workflows/release.yml`. Verify a downloaded release zip with:

```
gh attestation verify <zip> --repo gediz/teams-web-chat-exporter
```

This proves our CI built those exact bytes from that commit. It does not prove the store-installed CRX/XPI matches (the stores re-sign), and it does not vouch for the source itself. It is a layer, not a guarantee.

## Per-file hashes

Each release attaches `SHA256SUMS.txt` with the SHA-256 of every built zip. These survive store repackaging because you compare individual files, not the whole package. The postinstall-generated assets (the Twemoji pack and `hb-subset.wasm`) are not committed as binaries, but their hashes are pinned in `vendor-hashes.json` and checked in CI, so they are accounted for by public source and reproducible from the pinned dependencies.

## Reproduce the build

1. Clone the repo at the release tag.
2. Use the toolchain in `.nvmrc` (Node) and the `pnpm-lock.yaml` lockfile.
3. `pnpm install --frozen-lockfile`
4. `pnpm zip` (Chrome), `pnpm zip:firefox` (Firefox + sources), `pnpm zip:edge` (Edge).
5. Compare the per-file SHA-256 of the built output against `SHA256SUMS.txt` and the unpacked store package.

## Known limits today

- The zip step is not yet byte-deterministic (timestamps and file order), so reproducing the whole zip hash may differ. Compare per-file payload hashes, which are stable.
- pnpm is pinned exactly via the `packageManager` field (`pnpm@10.23.0`), and Node is pinned exactly via `.nvmrc` (`24.16.0`), which both workflows read. The GitHub Actions are pinned to full commit SHAs.
- The api-client credentialed-fetch guard currently logs a non-Microsoft host but does not block (warn-only); enforcement is planned for a later release.
- The guard checks the request URL. Server 30x redirects are still followed (the fetch default redirect mode), and a custom `authentication` header on the Teams Free path survives a cross-origin redirect, so a redirect from a Microsoft host to a non-Microsoft host is a known residual. It requires an attacker to control a Microsoft TLS response, and closing it (`redirect:'manual'` plus re-checking the location) is planned alongside guard enforcement, with real-tenant testing.
